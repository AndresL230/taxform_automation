# TaxExtract

Upload a U.S. tax document, get back structured, reviewable fields. TaxExtract
classifies a form, extracts its fields with a per-field confidence score and a
bounding box, and presents everything in a review UI that flags anything a human
should check before trusting it.

**Live demo:** https://taxform-automation.andreslopez-23061.workers.dev

The demo seeds a few example documents so you can explore the review UI without an
API key. Uploading your own file runs live extraction (the deployment has its own
key).

## Supported forms

| Form | Fields extracted |
| --- | --- |
| W-2 | wages, federal withholding, social security wages, employer EIN, employee SSN, employee name, employer name |
| 1099-NEC | nonemployee compensation, federal withholding, payer TIN, recipient TIN, payer name, recipient name |
| 1099-INT | interest income, early withdrawal penalty, interest on U.S. Savings Bonds and Treasury obligations, federal withholding, payer TIN, recipient TIN, payer name, recipient name |
| 1099-DIV | total ordinary dividends, qualified dividends, total capital gain distr., federal withholding, payer TIN, recipient TIN, payer name, recipient name |

Anything else classifies as unsupported and comes back failed with the message
`Detected {type}, not a supported form.`

Out of scope for now: 1099-B composites and Schedule K-1 (multi-page, hundreds of
transaction lines, high layout variance) and auth.

## How it works

Extraction is a two-pass pipeline over a **form registry**, so adding a form is
configuration rather than new code.

1. **Classify.** One cheap model call identifies the form type.
2. **Route.** The detected type is normalized and looked up in the registry
   (`src/extract/registry.ts`). An unknown type fails fast without a second call.
3. **Extract.** A second call uses that form's own response schema and prompt
   (the shared scaffold in `src/extract/prompt.ts` plus the form's field fragment).
4. **Build.** `buildDocument` joins the model output with the form's backend field
   constants and assigns a status:
   - `ready`: every field present and confident (confidence >= 0.7)
   - `needs_review`: any empty value or any field below 0.7
   - `failed`: document was not legible, or not a supported form

The model never invents the field constants (label, box, type); it only transcribes
values, with strict rules to prefer an empty, low-confidence answer over a guess.
Currency is normalized to digits and a single decimal point; SSNs and EINs are kept
as printed, masking preserved.

The extraction model is Gemini (`gemini-3.5-flash`) via `@google/genai`, with Zod
validating every response.

On the review page the actual document is rendered next to the fields: PDFs are
drawn with pdf.js (loaded lazily, only when a PDF is viewed) and images render
directly, and clicking a field highlights its bounding box on the document.

### Adding a form

A new form is a `FormDefinition` in `src/extract/<form>.ts` (field constants, a
prompt fragment, and schemas generated by `buildFormSchemas`) plus one entry in
`FORM_REGISTRY`. The shared join, prompt scaffold, status rules, dispatcher, and UI
are reused unchanged. `src/extract/int.ts` and `src/extract/div.ts` are the
canonical examples. See `docs/superpowers/specs/` and `docs/superpowers/plans/` for
the design and implementation notes.

## Project layout

```
src/
  extract/        form registry, classify-then-extract dispatcher, per-form definitions
    registry.ts   FORM_REGISTRY, getFormDefinition, normalizeFormType
    extract.ts    extractDocument: classify -> route -> extract -> build
    build.ts      buildFormSchemas, buildDocument (join + status), shared field schema
    prompt.ts     shared extract scaffold + classifier prompt
    w2.ts nec.ts int.ts div.ts   one FormDefinition per supported form
  api/            Cloudflare Worker API (router + POST /api/documents)
  components/     review UI (upload, table, field rows, confidence, status, badges)
  pages/          Home and Review routes
  state/          DocumentsContext (client-side document state)
  fixtures/       seed ExtractionResult JSON for the demo (one per form type)
  worker.ts       Worker entry: /api/* -> API, everything else -> static assets
scripts/
  capture-fixtures.ts   regenerate demo fixtures from the live API
  eval/                 form-parameterized accuracy harness (see its own README)
docs/superpowers/        specs and implementation plans
```

## API

A single stateless endpoint. No database; the client holds document state.

`POST /api/documents` with `multipart/form-data` and a `file` field
(`application/pdf`, `image/png`, or `image/jpeg`).

Response is an `ExtractionResult`:

```json
{
  "status": "ready",
  "detectedFormType": "1099-INT",
  "fields": [
    {
      "key": "interestIncome",
      "label": "Interest income",
      "box": "1",
      "value": "1284.55",
      "originalValue": "1284.55",
      "confidence": 0.98,
      "type": "currency",
      "bbox": { "page": 1, "x": 55, "y": 22, "w": 18, "h": 4 }
    }
  ]
}
```

Status codes: `200` (extraction ran, see `status`/`error` in the body), `400`
(missing or malformed `file`), `415` (unsupported content type), `405` (method
other than POST), `404` (unknown `/api/*` path).

## Getting started

Requires Node and npm.

```bash
npm install
```

### Develop

```bash
npm run dev      # Vite dev server, fast UI iteration (demo fixtures render)
npm run cf-dev   # full stack: builds, then runs the Worker via wrangler (live API)
```

`npm run dev` serves the frontend only, so the demo runs off seeded fixtures and
live upload is not available. Use `npm run cf-dev` to exercise the real
`/api/documents` endpoint locally. That needs a Gemini key (see Configuration).

### Test

```bash
npm test         # vitest run (unit suite, no network, no API key)
```

### Build and deploy

```bash
npm run build    # tsc -b && vite build  -> dist/
npm run deploy   # build, then wrangler deploy to Cloudflare Workers
```

The Worker serves the built SPA from `dist/` and handles `/api/*`.

## Configuration

The only secret is `GEMINI_API_KEY`.

- **Local:** copy `.dev.vars.example` to `.dev.vars` (gitignored) and set the key.
- **Production:** `wrangler secret put GEMINI_API_KEY`.

Cloudflare account and worker name are pinned in `wrangler.jsonc`.

## Evaluation harness

`scripts/eval/` is a standalone, form-parameterized accuracy harness that renders
synthetic IRS forms, degrades them, runs them through the same production extraction
path, and scores field by field. It is not part of the unit suite and needs your own
`GEMINI_API_KEY` and the official IRS PDFs. See `scripts/eval/README.md` for setup
and the `FORM=` switch.

## Tech stack

React 18, Vite, TypeScript, Tailwind CSS, React Router, Zod, and the Google GenAI
SDK, deployed on Cloudflare Workers. Tests run on Vitest.
