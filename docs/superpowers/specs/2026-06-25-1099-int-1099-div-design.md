# 1099-INT and 1099-DIV support, design

Date: 2026-06-25

## Goal

Add 1099-INT (Interest Income) and 1099-DIV (Dividends and Distributions) as the
third and fourth supported form types. The form registry, the classify-then-extract
dispatcher, and 1099-NEC already shipped (see
`2026-06-25-form-registry-1099nec-design.md`). This work is the first real test of
the registry's promise: each new form is **two FormDefinition entries plus their
schemas / promptFragments / fixtures**, not new pipeline code.

Three outcomes define done:

1. W-2 and 1099-NEC keep working identically (unchanged unit tests, unchanged eval).
2. 1099-INT and 1099-DIV work at the same quality bar (same anti-hallucination and
   value-format rules, via the COMMON scaffold).
3. The change touches no shared plumbing beyond the registry's own config surface.

## Leak test (the explicit guard)

The brief says: if this requires new join logic, a new status code, or new route
handling rather than two `FormDefinition` entries, STOP and report a leaking
abstraction. It does not. Confirmed by reading the current code:

- `buildFormSchemas(fieldKeys)` (`src/extract/build.ts`) generates both the Gemini
  `responseSchema` and the Zod `validate` from a list of field keys. Reused as-is.
- `buildDocument(parsed, formDef)` (`src/extract/build.ts`) does the join and assigns
  status (`failed` / `needs_review` / `ready`) driven by `formDef.fieldDefs`. Reused
  as-is. No new status code.
- `buildExtractPrompt(formDef)` (`src/extract/prompt.ts`) is the COMMON scaffold
  (anti-hallucination rules, VALUE FORMATTING, CONFIDENCE tiers, BOUNDING BOXES) with
  `formDef.promptFragment` spliced in. Reused as-is.
- `extractDocument` (`src/extract/extract.ts`) classifies, routes via
  `getFormDefinition`, extracts, and builds. Reused as-is. No new route handling.
- The bbox normalization seam in `buildDocument` is shared and unchanged.

### The one edit to a shared file, and why it is not a leak

`src/extract/registry.ts` changes, but only its **config surface**:

- Two entries added to `FORM_REGISTRY` (`INT_FORM`, `DIV_FORM`).
- Two branches added to `normalizeFormType` for the no-dash spellings
  (`1099INT` -> `1099-INT`, `1099DIV` -> `1099-DIV`), exactly mirroring the existing
  `W2` and `1099NEC` branches.

`normalizeFormType` is a spelling whitelist, not pipeline logic; the canonical dash
spellings (`1099-INT`, `1099-DIV`) already route correctly through the registry key
alone, so these branches only add robustness for the no-dash variant. `getFormDefinition`,
`supportedFormTypes`, and the unknown -> `"Detected {type}, not a supported form."`
failure are all derived and untouched. This is config inside the registry module, the
same kind of change NEC made, not new plumbing.

`src/extract/prompt.ts`: the classifier prompt already lists `"1099-INT"` as a
recognized example; add `"1099-DIV"` to that same example string. Copy-only, no logic.

## Field definitions

Box numbers and labels confirmed against the current IRS instructions (i1099int and
i1099div, fetched 2026-06-25):

- 1099-INT: Box 1 "Interest income", Box 2 "Early withdrawal penalty", Box 3
  "Interest on U.S. Savings Bonds and Treasury obligations", Box 4 "Federal income
  tax withheld".
- 1099-DIV: Box 1a "Total ordinary dividends", Box 1b "Qualified dividends", Box 2a
  "Total capital gain distr.", Box 4 "Federal income tax withheld".

Both forms carry labeled (unnumbered) PAYER'S name / PAYER'S TIN / RECIPIENT'S TIN /
RECIPIENT'S name regions, handled identically to NEC: `box` is `''`, `payerTIN` typed
`ein`, `recipientTIN` typed `ssn`. There is no `tin` member of `FieldType`; the
brief's "ein/tin" and "ssn/tin" notation is descriptive of the TIN's usual format,
and reusing the existing `ein` / `ssn` types keeps normalization and the eval scorer
working unchanged. The narrow TIN-polymorphism limitation (a business recipient
carries an EIN-format TIN under an `ssn`-typed field; transcribed as printed) is the
same known, accepted limitation noted for NEC.

### Definition module shape

Each of `src/extract/int.ts` and `src/extract/div.ts` mirrors `src/extract/nec.ts`
exactly:

```ts
export const INT_FIELDS = [ ... ] as const satisfies readonly FieldDef[]
const INT_PROMPT_FRAGMENT = `...`
const intSchemas = buildFormSchemas(INT_FIELDS.map((f) => f.key))
export const INT_FORM: FormDefinition = {
  formType: '1099-INT',
  fieldDefs: INT_FIELDS,
  responseSchema: intSchemas.responseSchema,
  validate: intSchemas.validate,
  promptFragment: INT_PROMPT_FRAGMENT,
}
```

## Classification

`extractDocument` already classifies, then routes through `getFormDefinition`. With
`INT_FORM` / `DIV_FORM` registered and the two `normalizeFormType` branches added, a
detected `1099-INT` / `1099-DIV` (any common spelling) routes to the right definition.
Unknown types continue to fail with the existing
`"Detected {type}, not a supported form."` message (comma, no dash). No dispatcher
change.

## UI and types

No frozen-type change. `Document.formType` is already `string`; `applyExtraction`
already sets `formType: result.detectedFormType`; `FormTypeBadge` is a pure
passthrough of `formType`; `FieldRow` already hides the box sub-label when `box` is
empty (added for NEC's unboxed identity fields). A document detected as `1099-INT` or
`1099-DIV` renders its badge and its 8 fields with zero component changes. Nothing in
the UI hardcodes a form list. If implementation turns up any hardcoded form list, that
is a leak to flag, not to work around.

## Tests (unit, mocked SDK, green suite)

### New

- `src/extract/int.test.ts`: mirror `nec.test.ts`. `buildDocument` with a mocked INT
  `ParsedExtraction` -> 8 `Field`s in the frozen shape, in `INT_FIELDS` key order;
  status tiers correct (`ready`; `needs_review` on an empty value or `confidence < 0.7`;
  `failed` on `!isLegible`).
- `src/extract/div.test.ts`: same, for `DIV_FORM` (8 fields, DIV key order, status
  tiers).
- Extend `src/extract/registry.test.ts`: `normalizeFormType` canonicalizes the INT/DIV
  spellings (`1099-int`, `1099 INT`, `1099-div`, `1099 DIV`); `getFormDefinition`
  returns the INT and DIV defs; `supportedFormTypes` contains all four.

### Routing

`src/extract/extract.test.ts` mocks the SDK and branches on prompt text (classify vs
extract), with one explicit case per form (W-2, NEC) today. Add two matching cases:
set `state.classifyType` to `'1099-INT'` / `'1099-DIV'` with a full 8-field
`extractPayload`, and assert the result routes to the right def, is `ready`, reports
the detected type, and returns the 8 fields in `INT_FIELDS` / `DIV_FIELDS` key order.
The existing unsupported-type case (`1098` -> `failed`, never calls extract) stays.

### Regression (unchanged, prove no break)

All existing W-2 and 1099-NEC tests pass unchanged: `w2.test.ts`, `nec.test.ts`,
`build.test.ts`, `extract.test.ts` (existing cases), `documents.test.ts`,
`worker.test.ts`, and the eval `*.test.ts`. `fixtures.test.ts` updates only to account
for the two added fixtures and their field counts (see below); existing W-2/NEC

## Fixtures and eval (seed now, user captures/runs live later)

### Seed fixtures

- `src/fixtures/int.json` and `src/fixtures/div.json`: post-build `ExtractionResult`
  JSON (`status: 'ready'`, `detectedFormType: '1099-INT'` / `'1099-DIV'`, 8 fields each
  in the frozen `Field` shape with plausible obvious-fake values and in-range bboxes),
  mirroring `nec.json`.
- Register both in `src/fixtures.ts` as new entries (reusing the existing sample asset
  for `fileUrl`, a placeholder the user overwrites via capture), so the demo shows all
  four form types.
- `src/fixtures.test.ts`: update the document count and make field-count assertions
  form-aware (INT and DIV ready docs have 8 fields). No change to the W-2/NEC/failed
  assertions.

### Capture script

`scripts/capture-fixtures.ts`: add commented INT and DIV `SAMPLES` entries
(`out: 'int'` / `'div'`, `form: '1099-INT'` / `'1099-DIV'`) for the user to fill in
with their assets. It already calls the production `extractDocument`, so captured
output is byte-identical to the server's.

### Eval harness (already form-parameterized)

The harness was generalized for NEC: `run.ts` selects the form via `FORM=`,
`getEvalForm` looks up an `EvalForm` from `EVAL_FORMS` in `scripts/eval/forms.ts`, and
the scorer/degrade axes are form-agnostic. To cover INT/DIV add, mirroring the NEC
scaffold:

- `scripts/eval/groundtruth-int.ts` and `groundtruth-div.ts`: an `INT_SCORED_KEYS` /
  `DIV_SCORED_KEYS` list (in field order), a `*FormData` type, and a `make*Scenario`
  generator with the shared `clean` / `zero_withholding` / `masked_tin` /
  `large_values` scenarios, obvious-fake EIN/SSN helpers, and currency
  comma-printed/strip-expected handling (reuse the NEC helpers' approach).
- `scripts/eval/groundtruth-int.test.ts` and `groundtruth-div.test.ts`: mirror
  `groundtruth-nec.test.ts` (all scored fields in order; obvious-fake TINs; blank box
  expects empty; masked TIN preserved).
- Two `EvalForm` entries in `EVAL_FORMS` (`'1099-INT'`, `'1099-DIV'`) with `asset`
  (`f1099int.pdf` / `f1099div.pdf`), `scenarios`, `scoredKeys`, `seeds`, and a
  best-guess AcroForm `fieldMap` carrying the existing NEC-style note to reconcile
  against the real PDF via `DUMP_FIELDS`.

Scaffold only: the user supplies the live `GEMINI_API_KEY` and the `f1099int.pdf` /
`f1099div.pdf` assets and runs make + degrade + score with `FORM=1099-INT` /
`FORM=1099-DIV`.

## Out of scope

- 1099-B composites and Schedule K-1 (the hard tier), explicitly deferred. Rationale:
  these are multi-page, carry hundreds of transaction lines (1099-B), and have high
  layout variance across issuers and consolidated statements, so they do not fit the
  one-page, fixed-field FormDefinition shape that makes INT/DIV pure config. They need
  their own design (line-item extraction, pagination, per-lot schemas), not a registry
  entry.
- PDF rendering in the document viewer (F1) and auth.

## Decisions log

- INT/DIV are pure config: a `src/extract/<form>.ts` definition plus a registry entry
  each. No edits to `build.ts`, `prompt.ts` (beyond one classifier example string), or
  `extract.ts`.
- `payerTIN` -> `ein`, `recipientTIN` -> `ssn`, reusing the existing `FieldType` set.
  No new `tin` type; the brief's "ein/tin" / "ssn/tin" is descriptive.
- Box labels taken verbatim-equivalent from the current IRS i1099int / i1099div
  instructions (fetched 2026-06-25), in the codebase's sentence-case label style.
- `normalizeFormType` gains two spelling branches (config), mirroring the NEC
  precedent; flagged explicitly as the only edit to a shared file, and judged not a
  leak.
- Eval `fieldMap`s for INT/DIV are best-guess AcroForm names, reconciled live via
  `DUMP_FIELDS`, consistent with how NEC's map was seeded.
